Phase 3 Local Keycloaks for proper user mgmt

Federation, with the governance tier preserved

Each participant runs its own Keycloak. The Authority KC stays in place as the dataspace's federation point — it brokers to each local KC via OIDC instead of holding participant users.

What's new
  • Per-participant Keycloak (e.g. realm caney-fork)
  • One IdP entry per participant on Authority KC
  • KC_IDP_HINT-driven brokering
  • Catalog UI does a two-tier OIDC flow
What stays the same
  • EDC machinery (policies, claims, DSP trust)
  • Connector-to-connector via client_credentials
  • oauth2-proxy + X-Api-Key at the edge

Phase 3 Two-tier login sequence

Phase 3 Why two tiers?

The naive simpler model — "Catalog UI logs into the local KC directly" — works, but it loses two things:

  1. Federated audit at the governance layer. Authority KC can no longer see which orgs are participating in which DSP exchanges.
  2. A single SSO entry point for governance tooling — onboarding dashboard, contract-management UI, certification status — all of which live on the governance VM.

The two-tier flow keeps both properties while delegating user mgmt to the participant that actually owns the org.
The silent participant-KC step is specifically so the /management/ edge can be locked down by an oauth2-proxy that the participant operator controls.

Phase 4 OID4VCI / OID4VP

The long-term direction

Authority Keycloak (and the per-participant KCs from Phase 3) step out of the runtime trust path.
Users hold Verifiable Credentials in wallets. Connectors hold VCs in their Identity Hubs. Verifiers check signatures + a trust list of issuer DIDs.

The road there is mostly replacement, not addition:

  • Authority KC → OID4VCI issuer (Walt.id / KC plugin / custom)
  • glcdi-iam-keycloak → iam-identity-trust in EDC
  • Catalog UI OIDC client → Catalog UI VP verifier
  • Per-participant KCs → removed, or repurposed as participant-local issuers

Phase 4 Why long-term?

The IDENTITY.md analysis covers why OIDC — not OID4VC — is right for the prototype today:

  • OID4VC specs are still Implementer's Drafts — breaking changes between drafts are common
  • The wallet ecosystem (EUDI Wallet, etc.) is pilot-stage, not production-stage
  • Trust frameworks (EUDI ARF, Gaia-X Trust Framework) are still firming up

Phase 4 here is the operational shape when those preconditions are met.
The point of writing it down now is that Phases 1–3 stay compatible with this end-state: the claim shape and the EDC policy functions survive intact.

Takeaways

  1. Phase 1 is real and shipped. Authority KC, the onboarding form, the admin dashboard, the realm-admin SA, the SMTP wiring — all live in governance-services/ and on the staging VM.
  2. Each next phase is additive. Phase 2 lifts the X-Api-Key floor by adding user OIDC on top. Phase 3 federates it. Phase 4 swaps the issuer for VCs.
  3. The claim shape is the load-bearing design choice. Because glcdi_organization, glcdi_roles, etc. are the same string in every phase, the EDC-side policy functions and the access-control rules never need a rewrite.
  4. Operational complexity grows then shrinks. P1 is simple. P2 adds an OIDC client. P3 adds N participant KCs. P4 ultimately replaces all of them with a trust list of issuer DIDs.
